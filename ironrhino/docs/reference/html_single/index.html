序言

目标

简化web应用程序开发,提高可维护性

Ironrhino是在spring,struts2,hibernate等成熟的开源框架之上提供一些常用的功能,开发者可重用原有的技术积累和开发习惯,不需要重新学习一套新体系

Ironrhino提供如下核心功能:

  • 自动配置,简单的java annotaion和约定俗成的命名即可实现CRUD

  • 支持ajax,基于jquery,开发者可不需写js代码,只需在html tag上增加class和属性,就可以支持异步提交和浏览器端的UI增强(比如日期输入控件和tab等)

  • 用户管理,只需加入一个jar包即可实现用户登录注销,用户增删改,授权,再放入另外一个jar包可记录用户的登录行为,比如登录时间登录地点成功是否

  • 全文检索,基于compass(lucene之上的一个全文检索框架),在model类上加入compass的annotaion即可实现全文检索,支持增量建索引和重建索引

Ironrhino还提供如下高级功能:

  • session重写,session中的对象保存在加密cookie或者cache服务器之中,这样应用服务器可以水平扩展,重启后数据也不会丢失

  • remoting功能,加一个annotaion即可对外发布hessian和httpinvoker服务,引用remoting服务只需在spring里面简单配置指定接口即可,可使用zookeeper在集群和分布式环境下自动发布和发现服务,并支持故障转移

  • aop缓存,在获取对象的方法上加annotaion可以对返回结果缓存,支持设置超时,同样可以在修改或者删除对象的方法上加annotaion通知清除缓存,这样可以保证不出现脏数据

  • 数据路由,在数据访问方法上加annotaion可以指定访问哪个数据源

  • 分布式文件存储,可以使用hadoop的hdfs作为文件存储系统

Ironrhino还提供如下小功能:

  • 内置了一个console,可以调用spring bean的方法,可以修改bean的属性并且会通过jms更新到其他集群机器

  • 在struts2 action的方法上加annotation支持captcha功能(验证码)和防csrf(跨站点引用)攻击

  • 异步发送邮件,当jms消息出问题时会自动切换到使用同步式发送

  • 内置一个统计器,可以记录页面访问量,交易笔数,交易金额等,并且配有图表展示

  • 内置一个简单的cms管理系统

感谢Spring和Struts2和Hibernate和Compass和JQuery等团队!

核心功能

Chapter 1. Introduction

1.1. Overview

Compass API

As you will learn in this chapter, Compass high level API looks strangely familiar. If you used an ORM framework (Hibernate, JDO or JPA), you should feel right at home. This is of-course, intentional. The aim is to let the developer learn as less as possible in terms of interaction with Compass. Also, there are so many design patterns of integrating this type of API with different applications models, that it is a shame that they won't be used with Compass as well.

For Hibernate users, Compass maps to SessionFactory, CompassSession maps to Session, and CompassTransaction maps to Transaction.

Compass is built using a layered architecture. Applications interacts with the underlying Search Engine through three main Compass interfaces: Compass, CompassSession and optionally CompassTransaction. These interfaces hide the implementation details of Compass Search Engine abstraction layer.

Compass provide access to search engine management functionality and CompassSession's for managing data within the Search Engine. It is created using CompassConfiguration (loads configuration and mappings files). When Compass is created, it will either join an existing index or create a new one if none is available. After this, an application will use Compass to obtain a CompassSession in order to start managing the data with the Search Engine. Compass is a heavyweight object, usually created at application startup and shared within an application for CompassSession creation.

CompassSession as the name suggests, represents a working lightweight session within Compass (it is non thread safe). With a CompassSession, applications can save and retrieve any searchable data (declared in Compass mapping files) from the Search Engine. Applications work with CompassSession at either the Object level or Compass Resource level to save and retrieve data. In order to work with Objects within Compass, they must be specified using either OSEM, XSEM (with XSEM XmlObject) or JSEM. In order to work with Resources, they must be specified using RSEM (Resource can still be used with OSEM/XSEM/JSEM to display search results, since Objects/Xml/Json end up being converted to Resources). Compass will then retrieve the declared searchable data from the Object automatically when saving Objects within Compass. When querying the Search Engine, Compass provides a CompassHits interface which one can use to work with the search engine results (getting scores, resources and mapped objects).

Applications work with CompassSession at either the Object level or Compass Resource level to save and retrieve data. In order to work with Objects within Compass, they must be specified using either OSEM, XSEM (with XSEM XmlObject) or JSEM. In order to work with Resources, they must be specified using RSEM (Resource can still be used with OSEM/XSEM/JSEM to display search results, since Objects/Xml/Json end up being converted to Resources). Compass will then retrieve the declared searchable data from the Object automatically when saving Objects within Compass. When querying the Search Engine, Compass provides a CompassHits interface which one can use to work with the search engine results (getting scores, resources and mapped objects).

Compass API

As you will learn in this chapter, Compass high level API looks strangely familiar. If you used an ORM framework (Hibernate, JDO or JPA), you should feel right at home. This is of-course, intentional. The aim is to let the developer learn as less as possible in terms of interaction with Compass. Also, there are so many design patterns of integrating this type of API with different applications models, that it is a shame that they won't be used with Compass as well.

For Hibernate users, Compass maps to SessionFactory, CompassSession maps to Session, and CompassTransaction maps to Transaction.
